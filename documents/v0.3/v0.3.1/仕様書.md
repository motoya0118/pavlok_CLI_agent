## 改修後仕様

### 1. カロリー計算機能
1. `/cal` コマンドで食事画像アップロードモーダルを開く
2. モーダルは Slack `file_input` を使い、画像は 1 枚のみ受け付ける
   1. 対応拡張子は `jpg`, `jpeg`, `png`, `webp`, `heic`
   2. Slack アプリには `files:read` scope が必要
3. 画像サイズ上限は 10MB とする
   1. Slack `files.info` の `size`
   2. ダウンロード後バイト数
   3. 上記のどちらでも 10MB 超過なら失敗とする
4. モーダル submit 時は即時 ACK し、重い処理は非同期ジョブで継続する
5. 画像解析は `.env` の `CALORIE_PROVIDER=openai|gemini` で切り替える
   1. `openai`: `OPENAI_API_KEY`, `CALORIE_OPENAI_MODEL` を利用
   2. `gemini`: `GEMINI_API_KEY`, `CALORIE_GEMINI_MODEL` を利用
6. LLM からは `calorie_v1` 形式の JSON を期待するが、実装はレスポンス中の最初の JSON object を抽出して解釈する
7. 保存前に以下の正規化を行う
   1. `items` が空なら解析失敗
   2. `food_name` が空または `null` なら `不明`
   3. `calorie` は `int` 変換必須
   4. `calorie < 0` は解析失敗
8. 解析成功時は `/cal` 実行チャンネルに実行ユーザーへのメンション付きで通知する
   1. 通知内容は `カロリー解析結果を記録しました`
   2. Block Kit で品目別 kcal、合計 kcal、アップロード時刻(JST)を表示する
9. 解析失敗時の通知文言は以下の通りとする
   1. 画像未選択: モーダルエラー `画像を1枚選択してください。`
   2. 10MB 超過: `画像サイズが大きすぎるので、10MB以下の画像サイズにしてリトライしてください`
   3. `items=[]` など食事画像として成立しない場合: `upload画像はカロリー算出不可です。`
   4. その他の JSON/画像解析失敗: `画像解析に失敗しました`
   5. その他の実行失敗: `失敗しました。もう一度お試しください`

#### 1.1 DB設計（カロリー計算）
1. `calorie_records` テーブルを利用する
   1. `id` (uuid, PK)
   2. `user_id` (string, index)
   3. `uploaded_at` (datetime, JST の naive datetime, index)
   4. `food_name` (text, not null)
   5. `calorie` (int, not null, `>= 0`)
   6. `llm_raw_response_json` (text, not null)
   7. `provider` (string, `openai|gemini`)
   8. `model` (string, null 許容)
   9. `created_at` (datetime)
2. 複数品目は 1 品目 1 レコードで保存する
3. `llm_raw_response_json` は LLM から抽出した JSON 文字列をそのまま保存する

#### 1.2 LLM JSON契約 / Agent設計（カロリー計算）
1. 現在の期待 JSON は以下
```json
{
  "schema_version": "calorie_v1",
  "items": [
    {
      "food_name": "鶏むね肉",
      "calorie": 320
    }
  ],
  "total_calorie": 320
}
```
2. 共通 IF は `analyze_calorie(image_bytes, mime_type)` とする
3. 実装は OpenAI / Gemini の両方をサポートし、戻り値は以下
   1. `parsed_payload`
   2. `raw_json`
   3. `provider`
   4. `model`

### 2. レポート機能
1. `/config` でレポート配信デフォルト値を設定する
   1. 設定キーは `REPORT_WEEKDAY`, `REPORT_TIME`
   2. デフォルトは `sat`, `07:00`
   3. 値は `configurations` テーブルの既存 `key/value` 方式で保持する
   4. `REPORT_WEEKDAY` は `sun|mon|tue|wed|thu|fri|sat`
   5. `REPORT_TIME` は `HH:MM`
   6. 保存時の `value_type` は `STR`
2. `/config` の値はレポート入力 UI のデフォルト値として使う
3. 既存ユーザーには migration で `REPORT_WEEKDAY=sat`, `REPORT_TIME=07:00` を UPSERT する

#### 2.1 plan UI 連携
1. レポート入力 UI は既存 `plan` モーダル内に追加表示する
2. 表示条件は以下のいずれか
   1. `/plan` 実行日が `REPORT_WEEKDAY` と一致する
   2. 先月分 monthly レポートが未配信である
3. `先月分 monthly レポートが未配信` の判定は `report_deliveries` に以下の行が存在しないこととする
   1. `report_type='monthly'`
   2. `period_start=先月1日`
   3. `period_end=先月末日`
   4. `posted_at IS NOT NULL`
4. レポート入力 UI の入力形式は `plan` と同じく `[今日, 明日] + 時刻` とする
5. レポート入力 UI の初期値は以下の優先順で決める
   1. 同一ユーザーの最新 `pending report` の `input_value`
   2. `/config` の `REPORT_WEEKDAY`, `REPORT_TIME`
6. `/plan` モーダル submit 時の保存ルール
   1. `PLAN` / `REMIND` の `pending|processing` は wash で `canceled`
   2. `REPORT` は wash 対象外
   3. レポート入力欄が表示されている場合のみ `event_type=report` を保存する
   4. 同一ユーザーに `pending report` があれば UPDATE、なければ INSERT
   5. `processing` / `done` の report は UPDATE 対象外
7. `/plan` 完了通知のレポート表示ルール
   1. レポート入力欄が表示され、かつ保存対象があった場合のみ `📊 レポート予定` を表示する
   2. レポート入力欄が非表示の submit では、既存 `pending report` や `processing report` があっても返信には表示しない
8. `schedules.input_value` には以下の JSON を保存する
```json
{
  "ui_date": "today",
  "ui_time": "07:00",
  "updated_at": "2026-03-08T07:00:00"
}
```

#### 2.2 report スケジュール実行
1. worker は `state=pending AND run_at<=now` の schedule を取得し、実行前に `processing` へ更新する
2. `event_type=report` は `scripts/report.py` を実行する
3. `scripts/report.py` は `state=processing` の report schedule のみ処理対象とする
4. 同一期間のレポートが既に配信済みなら、新規投稿せず対象 schedule を `done` にして終了する
5. 既存 `report_deliveries` と `schedule_id` が衝突する場合は、配信用の代替 `report` schedule を作成して投稿に使う
6. スクリプト失敗時は worker 共通処理で `failed` とし、再試行上限未満なら `pending` に戻して `run_at` を再設定する

#### 2.3 月次 / 週次判定
1. レポート種別判定は `report` 実行時点で行う
2. 判定ルール
   1. 先月分 monthly が配信済みなら `weekly`
   2. 未配信なら `monthly`
3. monthly が未配信の間は weekly を配信しない
4. 配信済み判定は `report_deliveries.posted_at IS NOT NULL` とする

#### 2.4 集計期間
1. monthly
   1. 先月1日から先月末日まで
2. weekly
   1. 前回 weekly の `period_end + 1日` から今回発行日の前日まで
   2. 前回 weekly が無い場合は当月1日から今回発行日の前日まで

#### 2.5 集計ロジック
1. 集計対象は以下の `REMIND` schedule のみとする
   1. `user_id` 一致
   2. `event_type=REMIND`
   3. `thread_ts IS NOT NULL`
   4. `run_at` の日付が集計期間内
2. 成功判定は `action_logs.result=YES` が 1 件でもあれば成功、それ以外は失敗とする
3. 集計単位はコミットメント単位とする
   1. 原則 `commitment_id` ごとに集約する
   2. `commitment_id` が無い既存データは `task 名` を fallback key に使う
4. 出力指標は以下
   1. 合計 `success_count`
   2. 合計 `failure_count`
   3. 合計 `success_rate`
   4. `by_commitment[]`
5. `by_commitment[]` の各行は以下
   1. `task`
   2. `success_count`
   3. `failure_count`
   4. `success_rate`
6. 対象 0 件の場合
   1. 合計値は `success_count=0`, `failure_count=0`, `success_rate=0.0`
   2. 表示用のコミットメント行は `（対象なし）` を 1 行生成する

#### 2.6 レポートコメント生成
1. 既存の Codex 実行機構を再利用してコメントを生成する
2. プロンプトテンプレートは `prompts/report_comment.md` を利用する
3. LLM 入力 payload は以下
```json
{
  "schema_version": "report_comment_v1",
  "report_type": "weekly",
  "period_start": "2026-03-01",
  "period_end": "2026-03-07",
  "stats": {
    "success_count": 5,
    "failure_count": 2,
    "success_rate": 71.4
  },
  "commitment_stats": [
    {
      "task": "運動する",
      "success_count": 3,
      "failure_count": 1,
      "success_rate": 75.0
    }
  ]
}
```
4. LLM の期待出力はコメント本文のみとする
5. Codex 実行失敗時は成功率ベースの fallback コメントを返す

#### 2.7 Slack 投稿形式
1. レポート投稿先は `plan/remind` と同じチャンネルとする
2. 投稿は Block Kit の card 形式とし、Markdown 表は使わない
3. 投稿ブロックの構成は以下
   1. 実行ユーザーメンション
   2. レポート種別と期間
   3. 集計サマリー
   4. コミットメントごとの 1 行サマリー
   5. `コーチコメント`
   6. `読みました` ボタン
4. コミットメントごとの表示は以下
   1. `• タスク名`
   2. `成功`
   3. `失敗`
   4. `成功率`
5. `report_deliveries.markdown_table` カラム名は据え置くが、現実装では Markdown 表ではなくプレーンテキスト要約を保存する
   1. 先頭に合計行
   2. 続けてコミットメント別の複数行

#### 2.8 read / ignore / Pavlok
1. report 投稿成功後、対象 schedule は `processing` を維持する
2. 投稿成功後にのみ `report_deliveries` を INSERT する
3. `report_deliveries.thread_ts` には親投稿の `ts` を保存する
4. `読みました` ボタンの `action_id` は `report_read`
5. 初回押下時は以下を実施する
   1. `action_logs.result=REPORT_READ` を INSERT
   2. `report_deliveries.read_at` を更新
   3. `schedule.state=done`
   4. 元レポートスレッドへ返信
   5. Pavlok の notification stimulus を送る
6. 返信文は種別別に以下
   1. weekly: `来週も頑張りましょう`
   2. monthly: `来月も頑張りましょう`
7. 2 回目以降の押下は後続無視とする
   1. 既に `REPORT_READ` があれば `すでに確認済みです。` を返す
   2. その際 `read_at` が欠落していれば backfill する
   3. `schedule.state` が `done` でなければ補正する
8. ignore 監視は既存機構をそのまま利用する
   1. 対象は `plan/remind/report`
   2. ユーザーごとに最新 `processing` 1 件のみ監視する

#### 2.9 DB設計（レポート）
1. 既存テーブル改修
   1. `schedules.event_type` に `report` を追加
   2. `schedules.input_value` を追加
   3. `ck_schedules_event_commitment_id` を以下へ更新
      `(upper(event_type) = 'REMIND' AND commitment_id IS NOT NULL) OR (upper(event_type) IN ('PLAN', 'REPORT') AND commitment_id IS NULL)`
   4. `action_logs.result` に `REPORT_READ` を追加
2. `report_deliveries` テーブル
   1. `id` (uuid, PK)
   2. `schedule_id` (FK -> schedules.id, UNIQUE)
   3. `user_id` (string, index)
   4. `report_type` (`weekly|monthly`)
   5. `period_start` (date)
   6. `period_end` (date)
   7. `posted_at` (datetime, not null)
   8. `read_at` (datetime, null 許容)
   9. `thread_ts` (string, null 許容)
   10. `markdown_table` (text, not null)
   11. `llm_comment` (text, null 許容)
   12. `created_at`, `updated_at`
   13. `UNIQUE(user_id, report_type, period_start, period_end)`

#### 2.10 migration 方針
1. Alembic 単一 revision で以下を実施する
   1. `report_deliveries` 作成
   2. `schedules` 再構築
   3. `action_logs` 再構築
   4. 既存ユーザーへ `REPORT_WEEKDAY`, `REPORT_TIME` を UPSERT
2. SQLite 運用を前提に再構築型 migration を採用する
