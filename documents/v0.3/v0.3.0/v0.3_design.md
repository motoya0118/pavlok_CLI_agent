# Oni System v0.3 Design Document

## 1. 概要

v0.3は「コマンド実行型」から「Slackコマンド/Webhookベース」への移行に伴うアーキテクチャの大幅な変更を伴うバージョンアップです。

### 主要な変更点

1. **Slack Integrationの変更**: スクリプト実行 → Webhookベースのイベント駆動
2. **commitment概念の追加**: 継続的なコミットメント管理
3. **plan/remindイベントフローの確立**: 24時間サイクルの計画・リマインド
4. **punishment機構の刷新**: scheduleテーブルベースの監視・実行
5. **Agent統合**: plan_updateスキルによるメッセージ生成

## 4. API設計 (FastAPI)

### 4.1 Slack Webhook Endpoints

#### POST /slack/command
Slackスラッシュコマンドを処理する。

**コマンド種別:**
- `/base_commit`: コミットメント登録UIを返却
- `/stop`: Punishment機構を停止
  - 実装: `SYSTEM_PAUSED`設定値を`true`に更新
  - Workerは各サイクル開始時に`SYSTEM_PAUSED`をチェックし、`true`の場合は処理をスキップ
- `/restart`: Punishment機構を再開
  - 実装: `SYSTEM_PAUSED`設定値を`false`に更新

#### POST /slack/interactive
Interactive Components (modal submit, button click) を処理する。

**アクション種別:**
- `commitment_submit`: コミットメント登録
- `plan_submit`: planタスクのやる/やらない選択
- `yes_button`: remindのYESボタン
- `no_button`: remindのNOボタン
- `config_submit`: 設定値保存
- `config_tab_change`: 設定モーダルタブ切り替え
- `config_reset_confirm`: 設定リセット確認

### 4.2 内部API (Workerから使用)

#### POST /internal/execute/{event_type}
Workerがスクリプト実行を通知する。

#### GET /internal/config/{key}
Workerが最新の設定値を取得する。
- キャッシュ戦略: Workerは60秒キャッシュを使用
- DB更新時の通知: 不要（ポーリング方式）

## 5. Worker設計

### 5.1 Punishment Worker

1分間隔で動作するバックグラウンドプロセス。

#### 実行ロジック
```
1. pending処理ループ:
   - schedulesテーブルから pending AND run_at <= now を取得
   - 1件ずつ processing に原子的更新 (SELECT FOR UPDATE)
   - event_type に応じたスクリプトをサブプロセス実行
   - 成功時:
     - plan: stateはprocessingのまま待機
     - remind: stateをdoneに更新
   - 失敗時: stateをfailed（retry_count < 3なら再スケジュールしてpending）
2. processing監視ループ（同一サイクル内で必ず実行）:
   - processing AND run_at <= now のplanを対象にignoreを検知
   - ユーザーごとに「最新のprocessing 1件のみ」を監視対象にする
3. 最新化ルール:
   - planを新規にprocessingへ上げる直前に、同一ユーザーの古いprocessing(plan)はcanceledへ更新
   - これにより罰の対象スレッドは常に最新のprocessingに一意化される
```

#### Ignore Mode検知
```
1. processing AND run_at <= now AND event_type=plan のレコードで検知
2. ユーザーごとに最新のprocessing 1件のみを対象にする
3. ignore_time = (now - run_at) // IGNORE_INTERVAL
4. punishmentsテーブルに (schedule_id, mode=ignore, count=ignore_time) が存在しない場合実行
5. アルゴリズム:
   - ignore_time == 1: vibe: 100
   - ignore_time > 1: zap: min(35 + (10 * (ignore_time - 2)), 100)
   - zap == 100: state → canceled, 行動ログ記録
```

### 5.2 Subprocess Scripts

#### scripts/plan.py
- BlockKit形式で24時間分の予定をSlackに投稿
- thread_tsを保存

#### scripts/remind.py
- BlockKit形式で激励メッセージ + YES/NOボタンを投稿
- thread_tsを保存

#### scripts/agent_call.py
- `echo {plan_prompt} | codex exec plan_update` を実行
- 結果をschedule.yes_comment/no_comment/commentに保存

## 7. フローデザイン

### 7.1 init フロー

```
User: /base_commit
  → Slack Modal表示 (commitment新規登録/編集)
User: Modal送信
  → Backend: commitmentsテーブル更新
  → Backend: 今日のplan eventがなければ登録
  → Slack: 完了通知
```

### 7.2 plan フロー

```
Worker: schedule.event_type=plan && pending && run_at <= now
  → 同一ユーザーの古いprocessing(plan)があれば state = canceled
  → state = processing
  → scripts/plan.py 実行
    → Slack: 「今日の予定を登録してください」＋ボタンを投稿
  → 待機...
User: ボタンをクリック
  → Slack: モーダルが開く（24時間分の予定入力UI）
User: 送信ボタン押下
  → Backend: plan_api処理（view_submissionイベント）
    → 対象plan: state = done
    → やる: remind eventをschedulesにINSERT
    → やらない: action_logsに記録
  → Slack: 登録完了通知（元メッセージを更新）
  → scripts/agent_call.py 実行
    → Agent: plan_updateで各レコードにコメント生成
    → schedule.yes_comment, no_comment, commentに保存
User: ×ボタンでモーダルを閉じる
  → 何もしない（stateはprocessingのまま、ignore検知で対応）
```

> **設計決定**:
> - モーダルはボタンクリックで開く（`trigger_id`が必要なため）
> - ×で閉じても再スケジュールしない（ignore検知で催促）

### 7.3 remind フロー

```
Worker: schedule.event_type=remind && pending && run_at <= now
  → state = processing
  → scripts/remind.py 実行
    → Slack: BlockKitで激励 + YES/NOボタンを新規投稿（スレッド返信はしない）
    → thread_tsを保存
  → 待機...
User: YESボタン
  → Backend: remind_api処理
    → action_logs: result=YES でINSERT
    → schedule: state = done
    → Slack: 新規メッセージでyes_commentを投稿
User: NOボタン
  → Backend: remind_api処理
    → action_logs: result=NO でINSERT
    → Pavlok API: NOモードで実行
    → Slack: 新規メッセージでno_commentを投稿
```

> **設計決定**: remindメッセージはplanのスレッドには返信せず、独立した新規投稿とする。
> 理由: スレッドが深くなるとSlack UIでの視認性が低下するため。

### 7.4 ignoreモード フロー

```
Worker: 1分間隔監視
  → processing AND run_at <= now AND event_type=plan を検査
  → ユーザーごとに最新processing 1件のみ対象化
  → ignore_time = (now - run_at) // IGNORE_INTERVAL
  → punishmentsテーブルで重複チェック
  → 存在しない場合:
    → ignore_time == 1: vibe: 100
    → ignore_time > 1: zap: min(35 + (10 * (ignore_time - 2)), 100)
    → punishmentsに記録
    → value == 100の場合: state = canceled, action_logs記録
```

### 7.5 NOモード フロー

```
User: NOボタン押下
  → action_logsから当日のNO_countを算出
  → 強度計算:
    - NO_count == 1: value = PAVLOK_VALUE_PUNISH（デフォルト35）
    - NO_count >= 2: value = min(PAVLOK_VALUE_PUNISH + (10 * (NO_count - 1)), LIMIT_PAVLOK_ZAP_VALUE)
  → punishmentsテーブルで重複チェック
  → Pavlok API実行（type=PAVLOK_TYPE_PUNISH, value=計算値）
```

> **設計決定**:
> - 初回NO時は`PAVLOK_VALUE_PUNISH`を使用（ユーザーが設定可能）
> - 2回目以降は10ずつ増加、最大`LIMIT_PAVLOK_ZAP_VALUE`でキャップ

| 設定キー | デフォルト | 型 | 範囲/有効値 | UI表示 | 説明 |
| -------- | --------- | --- | ---------- | ------ | ---- |
| `PAVLOK_TYPE_PUNISH` | zap | str | zap,beep,vibe | static_select | デフォルト罰スタイル |
| `PAVLOK_VALUE_PUNISH` | 35 | int | 0-100 | plain_text_input | デフォルト罰強度 |
| `LIMIT_DAY_PAVLOK_COUNTS` | 100 | int | 1-1000 | plain_text_input | 1日の最大ZAP回数 |

| `IGNORE_INTERVAL` | 900 | int | 300-1800 | static_select | ignore検知間隔(秒) |
| `IGNORE_JUDGE_TIME` | 3 | int | 1-30 | plain_text_input | ignore判定時間(秒) |

**`.env`で管理する項目（DB + Slack UI対象外）:**

| 設定キー | デフォルト | 型 | 範囲/有効値 | 説明 |
| -------- | --------- | --- | ---------- | ---- |
| `TIMEOUT_REMIND` | 600 | int | 60-3600 | リマインド応答タイムアウト(秒) |
| `TIMEOUT_REVIEW` | 600 | int | 60-3600 | 振り返り応答タイムアウト(秒) |
| `RETRY_DELAY` | 5 | int | 1-60 | 失敗時リトライ遅延(分) |

### 10.2 設定値取得ロジック

```python
# 優先順位:
# - TIMEOUT_REMIND / TIMEOUT_REVIEW / RETRY_DELAY: 環境変数 > デフォルト
# - その他: DB > 環境変数 > ハードコードされたデフォルト
def get_config(key: str, default=None):
    if key in {"TIMEOUT_REMIND", "TIMEOUT_REVIEW", "RETRY_DELAY"}:
        env_value = os.getenv(key)
        return parse_env_value(env_value) if env_value is not None else default

    # 1. DBから取得
    config = db.query(Configuration).filter_by(key=key).first()
    if config:
        return parse_value(config.value, config.value_type)

    # 2. 環境変数から取得
    env_value = os.getenv(key)
    if env_value is not None:
        return parse_env_value(env_value)

    # 3. デフォルト値
    return default
```

### 10.3 設定変更セキュリティ

1. **Slack署名検証**: 全エンドポイントで必須
2. **ユーザー認証**: `AUTHORIZED_USERS` でホワイトリスト管理
3. **入力バリデーション**: 型チェック、範囲チェック、有効値チェック
4. **危険な変更の確認**: ZAP強度80以上の場合は確認モーダル表示
5. **レート制限**: 同一パラメータへの変更は5分間に1回まで
6. **監査ログ**: 全変更を `config_audit_log` に記録

### 10.4 ロールバック機能

| レベル | コマンド | 動作 |
| ------ | -------- | ---- |
| 単一パラメータ | `/config rollback <key>` | 指定キーを直前の値に戻す |
| 全体ロールバック | `/config rollback --version N` | バージョンN全体を復元 |
| 緊急リセット | `/config reset` | 安全なデフォルト値に全リセット |

## 11. Slack UIデザイン詳細

> **詳細仕様**: 完全なBlockKit定義は [v0.3_slack_ui_spec.md](./v0.3_slack_ui_spec.md) を参照

### 11.1 UI設計方針

> 詳細は [v0.3_slack_ui_spec.md](./v0.3_slack_ui_spec.md) を参照

1. **モーダル活用**: 入力が必要な操作はモーダルで一元管理（`plain_text_input`はモーダル内でのみ使用可能）
2. **エフェメラル優先**: 個人向けの実行結果やエラー通知はエフェメラルメッセージで表示
3. **パブリックは通知のみ**: remind/ignore通知など、チーム全体に共有すべき情報のみパブリックメッセージ
4. **即時フィードバック**: ボタン操作に対して即座に応答

### 11.2 BlockKit制約への対処

BlockKitの `plain_text_input` はモーダル内でのみ使用可能。以下の戦略で対処：

| 入力タイプ | 解決策 |
|-----------|--------|
| 選択肢（列挙型） | `static_select` をモーダル内で使用 |
| 数値（範囲指定） | `plain_text_input` で入力 + バリデーション |
| ON/OFF | `checkboxes` をモーダル内で使用 |
| フリーテキスト | `plain_text_input` をモーダル内で使用 |

### 11.3 コマンド一覧

| コマンド | UI方式 | 表示タイプ |
| -------- | ------ | ---------- |
| `/base_commit` | BlockKit | エフェメラル |
| `/stop` | BlockKit (確認付き) | エフェメラル |
| `/restart` | BlockKit | エフェメラル |
| `/config` | BlockKit (タブ切り替え) | エフェメラル |
| `/config view` | BlockKit (一覧) | エフェメラル |
| `/config reset` | BlockKit (確認付き) | エフェメラル |
| `/config rollback <key>` | BlockKit | エフェメラル |
| `/audit` | BlockKit (履歴一覧) | エフェメラル |

### 11.4 Workerイベント通知

| イベント | UI方式 | 表示タイプ |
|----------|--------| ----------|
| plan投稿 | BlockKit (予定一覧 + スキップボタン) | パブリック |
| remind投稿 | BlockKit (YES/NOボタン) | パブリック |
| ignore検知 | BlockKit (応答催促) | パブリック |
| 応答完了通知 | BlockKit (スレッド返信) | パブリック |

### 11.5 設定変更UI構成

```
/config
  ├── タブボタン群 [罰設定] [Ignore] [Timeout] [詳細]
  ├── 現在値一覧
  │   ├── 設定項目 + [変更]ボタン
  │   └── ...
  └── アクション [リセット] [履歴]

[変更]クリック
  └── プリセット選択 OR +/- 調整
      └── [保存] [キャンセル]
```

### 11.6 Home Tab（代替UI）

App Homeタブを完全な設定管理UIとして提供：

- コミットメント管理（フォーム入力可能）
- 全設定の編集
- 統計ダッシュボード（今日の罰回数等）

**使い分け**:
- `/config` → 簡易確認・変更
- App Home → 詳細設定・一括編集

## 付録A: 設定管理機能のユースケース

### UC-1: 罰強度の調整

**ユーザー**: 「最近罰が甘すぎる気がする」

1. ユーザー: `/config` を実行
2. Slack: 罰設定タブ付きモーダルを表示
3. ユーザー: `デフォルト罰強度` を 50 → 70 に変更
4. Slack: "80以上は非常に強力です" という警告を表示
5. ユーザー: そのまま保存
6. Backend: `configurations` テーブル更新、`config_audit_log` に記録
7. Worker: 次回のサイクル(60秒以内)に新しい値を適用

### UC-2: 緊急時の罰停止

**ユーザー**: 「家族が来るから1時間だけ罰止めたい」

1. ユーザー: `/stop` を実行
2. Backend: Workerに停止シグナル送信
3. Slack: "罰機構を停止しました" を返信
4. 1時間後: ユーザーが `/restart` で再開

### UC-3: 設定の間違いに気づいてロールバック

**ユーザー**: 「やばい、IGNORE_INTERVALを60秒にしすぎた」

1. ユーザー: `/config rollback IGNORE_INTERVAL` を実行
2. Backend: `config_audit_log` から直前の値を取得
3. Backend: 元の値(900)で更新、監査ログ記録
4. Slack: "IGNORE_INTERVALを900に戻しました" を返信
